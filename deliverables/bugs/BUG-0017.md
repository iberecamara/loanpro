# Verifying a deleted user is no longer found by email fails in both environments for different reasons

## Description
This scenario verifies that once a user is deleted, looking up that user by email returns a 404 status code with a "User not found" message. The failure differs by environment. On prod, the delete request itself fails with a 401 status code using valid credentials, which prevents the test from reaching the follow-up lookup. On dev, the delete request succeeds, but the follow-up lookup by the deleted user's email returns a 500 Internal Server Error instead of the expected 404.

## Steps to Reproduce
1. Seed a user in the system so an existing record is available to delete.
2. Send a DELETE request to that user's email endpoint using valid credentials.
3. Observe the response status code of the delete request.
4. If the delete request succeeds, send a GET request to find a user using the deleted user's email address.
5. Observe the response status code and body of the lookup request.

## Actual Outcome
On prod, the delete request returns a 401 status code, so the user is not removed and the follow-up lookup cannot be verified.
On dev, the delete request succeeds, but the follow-up lookup returns a 500 status code with the body:
```json
{
    "message": "Internal server error"
}
```

## Expected Outcome
The delete request should return a 204 status code, and the follow-up lookup by the deleted user's email should return a 404 status code with the body:
```json
{
    "message": "User not found"
}
```

## Sample Payload
`DELETE /users/Ben.Jacobs@gmail.com`
```
Authorization: <valid token>
```
Followed by:
`GET /users/Jocelyn_Stanton@gmail.com`
