# Finding a user by their previous email after an update returns 500 instead of 404

## Description
When a user's email is updated, the user should no longer be found using their previous email address, and the API should return a 404 status code with a "User not found" message for that lookup. Instead, the API returns a 500 Internal Server Error. This also cascades into failures further down the test, where the response body for the new email lookup is the same 500 error object rather than user data, causing downstream assertions about the user's name and age fields to fail as well.

## Steps to Reproduce
1. Seed a user in the system so an existing record is available to update.
2. Send an update request for that user changing the name, email, and age to new values.
3. Send a GET request to find a user using the user's previous (pre-update) email address.
4. Observe the response status code and body.
5. Send a GET request to find a user using the user's new (post-update) email address.
6. Observe the response status code and body.

## Actual Outcome
The lookup by the previous email returns a 500 status code with the body:
```json
{
    "message": "Internal server error"
}
```
This same error body is also returned for the lookup by the new email, which then causes downstream checks on the user's name and age to fail because the response does not contain user data.

## Expected Outcome
The lookup by the previous email should return a 404 status code with the body:
```json
{
    "message": "User not found"
}
```
The lookup by the new email should return a 200 status code with the updated user data.

## Sample Payload
`PUT /users/Laverne30@yahoo.com`
```json
{
    "name": "Jerald McLaughlin",
    "email": "Eric_Kessler@gmail.com",
    "age": 106
}
```
Followed by:
`GET /users/Eric_Kessler@gmail.com`
